# 任务对话体验对比：Agent Workbench vs Codex 客户端

> Note: The `bash` tool has been removed. `shell` is now the only command execution tool. See `shell-unification-design.md`.

> 基于两张实际界面截图与 Codex 产品设计要点（见 `docs/best-agent-spec/codex-design.md`），对比**任务对话**层面的体验差距，并给出可落地的反思结论。  
> 与 `desktop-layout-reflection.md`（普通用户布局）互补：本文侧重**开发者/知识工作者**委托任务时的心智模型。

---

## 一、两张截图在说什么

### 1.1 Codex 客户端（左图）

```
┌─────────────┬──────────────────────────────────┬──────────────┐
│ New chat    │  按规范重构桌面客户端              │ Progress     │
│ Search      │  Worked for 8m 18s               │ ☑ 梳理组件…  │
│ Skills      │                                  │ ☐ 收敛 IA…   │
│             │  [自然语言计划：目标、步骤、约束]    │ Git          │
│ Projects ▾  │  Explored 2 files                │ main · 本地   │
│  agent-wb   │  Read App.tsx                    │ Sources      │
│   · 重构…22m│  Edited 1 file                   │              │
│             │  …                               │              │
│ Settings    │  Ask for follow-up changes  [5.5]│              │
└─────────────┴──────────────────────────────────┴──────────────┘
```

**一眼能读到的信息：**

- 我在做**哪件事**（任务标题 = 人话）
- 这件事**跑了多久**（Worked for 8m 18s）
- **做到哪几步了**（右侧 Progress 勾选）
- **动过哪些资产**（Explored / Read / Edited，一行一条，可扫读）
- **在哪个项目里**（Projects → agent-workbench）

### 1.2 Agent Workbench 客户端（右图）

```
┌──────────────────────────────────────────────────────────────┐
│ Agent Workbench   工作台 | Hub | 创建 Agent           🔔 👤  │
├──────────┬───────────────────────────────────────────────────┤
│ 对话      │ Tool Agent  ●本地已连接              [+ 新对话]  │
│ ls -l    │ 工作目录 /Users/.../myagent            [更换]     │
│ 你有哪些… │                                                   │
│ …        │  [用户] ls -l                                     │
│          │  [助手] ████ 完整终端 listing ████  @ bash         │
│          │  [用户] ?                                         │
│          │  [助手] 解释 rwx 权限…                            │
│          │  输入消息…                              📎 发送    │
└──────────┴───────────────────────────────────────────────────┘
```

**一眼能读到的信息：**

- 我在和**哪个 Agent 产品**说话（Tool Agent）
- **连接层**是否正常（本地已连接）
- **文件系统路径**是什么
- 左侧历史是**命令碎片**（`ls -l`）而非任务名
- 主内容区最显眼的是**终端原始输出**，不是「做到了什么」

### 1.3 一句话差距

| | Codex | Agent Workbench |
|---|--------|-----------------|
| 主对象 | **任务**（一件事的生命周期） | **对话**（一轮轮消息） |
| 主叙事 | 「正在帮你重构桌面端，已 8 分钟，完成 2/5 步」 | 「你发了 ls -l，这是终端输出」 |
| 执行过程 | 摘要行 + 右侧进度 | 原始日志塞进气泡 |
| 空间分工 | 左导航 / 中叙事 / 右状态 | 顶产品 Tab / 左会话 / 中聊天（全混在一起） |

## 二、分维度对比

### 2.1 信息架构：项目 → 任务 → 活动

| 维度 | Codex | Agent Workbench | 差距 |
|------|--------|-----------------|------|
| 组织轴 | **项目（仓库）** → 其下多条任务 | **全局会话列表**，按时间平铺 | 用户换项目时 AWB 无「归属感」 |
| 列表标题 | 「按规范重构桌面客户端」+ `22m` | 首条用户消息截断 → 常为 `ls -l` | 无法从侧栏恢复任务上下文 |
| 当前页标题 | 任务名（H1 级） | Agent 名 + 技术状态 | 屏幕中心标识的是「谁」不是「什么事」 |
| 二级信息 | Git、Sources、Progress 独立右栏 | 无右栏；Git/进度不存在于 UI | 工程型用户失去「监督面板」 |

**反思：** Codex 的侧栏是**任务索引**；AWB 的侧栏是**聊天记录索引**。后端虽有 `Session.title` 与 `status`，但标题生成策略偏「第一条消息」，UI 也未把 session 呈现为 task。

---

### 2.2 对话流：叙事流 vs 消息流

| 维度 | Codex | Agent Workbench | 差距 |
|------|--------|-----------------|------|
| 内容形态 | 长段**计划/说明**（目标、步骤、约束）与**活动行**交错 | 用户气泡 + 助手气泡，工具结果占大块 | 缺少「任务说明书」层 |
| 工具可见性 | `Explored 2 files`、`Edited 1 file` — **一行摘要** | 终端黑框、完整 stdout | 透明≠_dump 原始输出_ |
| 时间感 | `Worked for 8m 18s` 常驻标题区 | 仅消息时间戳 | 用户不知道还要等多久、是否仍在跑 |
| 后续输入 | `Ask for follow-up changes`（任务延续） | `输入消息…`（泛聊天） | 文案未强化「在同一件事上追问」 |

**反思：** 两者都「展示执行过程」，但 Codex 用**活动日志（activity feed）**服务监督感，AWB 用**聊天附件式终端**服务「证据感」。对监督者而言，前者扫一眼可知方向；后者需要读 20 行 listing 才知道 Agent 只是列了个目录。

当前 `ToolCallCard` 默认 `isExpanded: true`（running 时展开），与 Codex「默认折叠、主区保留叙事」相反。

---

### 2.3 任务生命周期：状态机 vs 开关

| 阶段 | Codex 界面线索 | Agent Workbench |
|------|----------------|-----------------|
| 创建 | New chat / 项目内新任务 | 工作台选 Agent → 模式弹窗 → 进聊天 |
| 执行中 | Worked for Xm、活动行滚动、Progress 未勾完 | `sending` 时 typing dots；无步骤进度 |
| 等待用户 | Follow-up 输入框、可能审批 | Bash 原生弹窗（仅 bash） |
| 可审查 | Progress、Sources、Git 右栏 | 无 diff/变更汇总视图 |
| 完成 | Progress 全勾、任务沉到列表下方 | `Session.status` 有 completed，** UI 未区分** |

**反思：** AWB 在数据模型上仍是「会话 ACTIVE」，不是「任务 RUNNING → REVIEW → DONE」。用户心理上是委托 Codex **办完一件事**；在 AWB 上是**和 Agent 来回聊**，完成边界模糊。

---

### 2.4 布局与注意力（与 Codex 截图对照）

| 区域 | Codex 职责 | AWB 职责 | 问题 |
|------|------------|----------|------|
| 顶栏 | 极简或融入窗口 | 产品级 Tab（工作台/Hub/创建） | **打断任务心流**，像切换网站 |
| 左栏 | 项目 + 任务列表 + 全局操作（Search/Skills） | 仅「对话」+ Agent 名重复 | 缺项目维度；缺 Search |
| 中栏 | **任务叙事 + 活动摘要**（主舞台） | 聊天气泡 + 终端（主舞台被日志占据） | 主次颠倒 |
| 右栏 | Progress / Git / Sources | **无** | 进度与仓库上下文无处可放 |
| 底栏 | 任务向 follow-up + 模型/权限 | 通用聊天输入 | 未表达「继续本任务」 |

Codex 三栏**各司其职**；AWB 两栏 + 顶栏，且中栏混入了 header、workspace、连接状态、消息、工具，**没有一栏只负责「状态与进度」**。

---

### 2.5 输入与权限

| 维度 | Codex | Agent Workbench |
|------|--------|-----------------|
| 权限表达 | `Full access` 等开关在输入区旁 | `本地已连接` + bash 弹窗 |
| 模型选择 | 输入区旁 `5.5 High` | 绑定在 Agent 配置，对话中不可见不可改 |
| 语义 | 「对当前任务的后续修改」 | 「又发一条消息」 |

**反思：** Codex 把**信任边界**放在用户即将下达指令处；AWB 把**传输层状态**放在标题旁。用户关心的是「它能动我电脑到什么程度」，不是「WebSocket 连没连上」。

## 三、对照 Codex 15 条设计要点：我们差在哪

| # | Codex 要点 | AWB 现状 | 任务对话相关差距 |
|---|------------|----------|------------------|
| 1 | 聊天 → 委托任务 | Session ≈ 聊天线程 | 无任务实体、无任务标题策略、无任务首页 |
| 2 | 真实工作流 | 后端工具有 bash/read/write | UI 未形成「读→改→测」叙事，只有离散 tool 卡片 |
| 3 | 多入口统一 | 桌面 + 管理后台 | 尚可，但桌面内路径仍分裂 |
| 4 | 异步后台任务 | 必须守着 SSE | 不能「交任务后走开」，侧栏无进行中标记 |
| 5 | 多 Agent 并行 | 单会话单 Agent | 侧栏未显示多个并行任务状态 |
| 6 | 透明执行过程 | 有 ToolCallCard | **展示形态错误**：RAW > 摘要 |
| 7 | 权限与审批 | 仅 bash 弹窗 | 无策略、无审批队列、右栏无待批事项 |
| 8 | 隔离环境 | workspace 字符串 | 路径条暴露技术细节，无 sandbox/变更审查感 |
| 9 | 可验证结果 | 测试/构建摘要 | 无「结果验收」区，只有 assistant 文字 |
| 10 | 团队标准 | Skills 文档化 | LoadSkill 有，对话内不可见「用了哪些 skill」 |
| 11 | 自动化 | 无 | 暂不比 |
| 12 | 复杂能力简单交互 | 自然语言入口有 | 前置步骤多（Agent/模式/目录） |
| 13 | 降低上下文负担 | Agent 自动探索 | 探索过程以终端形式呈现，加重负担 |
| 14 | 团队效率 | Hub/协作弱 | 任务对话未体现 review/PR |
| 15 | 人保留判断权 | 意图有 | 缺 Progress + diff 审查，用户只能读日志猜 |

## 四、深层反思：不是「少做了功能」，而是「对象建错了」

### 4.1 我们把 Session 当成了 Chat Room

Codex 的一屏对应：

```
Task = {
  title: "按规范重构桌面客户端",
  project: "agent-workbench",
  elapsed: "8m 18s",
  steps: [ { name, done } ],
  narrative: "...",
  activities: [ { type, target, at } ],
  artifacts: { git, sources }
}
```

AWB 的一屏对应：

```
Session = {
  agentName: "Tool Agent",
  messages: [ user, assistant+terminal ],
  workspace: "/Users/.../myagent",
  wsConnected: true
}
```

用户记忆的是 **「我正在重构桌面端」**（任务），不是 **「我正在和 Tool Agent 说话」**（频道）。侧栏标题应是前者；主区首屏应回答「进展如何」，而不是「上一句命令输出是什么」。

### 4.2 「透明」被误解为「把 stderr 贴进聊天」

Codex 截图里的透明执行：

- `Explored 2 files` — 数量 + 动作类型  
- `Read App.tsx` — 一个文件名  
- `Edited 1 file` — 结果规模  

这是 **activity line**，类似 CI 日志的折叠摘要，点击可展开详情。

AWB 的透明执行：

- 完整 `ls -l` 输出 — 与任务目标无关的细节占满视口  

**反思：** 任务对话 UX 里，执行过程应服务两个问题：「还在跑吗？」「方向对吗？」—— 摘要行 + 右栏 Progress 即可回答；原始终端应进二级面板，且默认折叠。

### 4.3 三栏不是炫技，是「三种阅读速度」

| 阅读速度 | Codex 落点 | AWB 落点 |
|----------|------------|----------|
| 快扫（3 秒） | 任务标题 + Worked for + Progress 勾选 | Agent 名 + 连接灯 + 路径 |
| 细读（1 分钟） | 中间计划段落 | 助手 Markdown/文字 |
| 深挖（需要时） | 点开 Read/Edited、Git、Sources | 滚终端、找 tool 卡片 |

AWB 把**深挖层**放在默认视口，把**快扫层**删掉或藏在技术字段里，导致与 Codex 的「差距感」远大于实际后端能力差距。

### 4.4 产品入口与任务入口打架

Codex 左栏顶部：**New chat / Search / Skills** — 全是「继续工作」的动作。  
AWB 顶栏：**工作台 / Hub / 创建 Agent** — 全是「管理产品」的动作。

用户在任务中途抬头，Codex 问的是「要不要开个新任务、搜旧任务」；AWB 问的是「要不要去 Hub 装个 Agent」—— **频道切走了，任务还在原地**。

任务对话体验要求：**任务内导航 >> 产品导航**。顶栏应退场或收成菜单。

### 4.5 截图中 Codex 也在谈「收敛三列」—— 给我们的启示

左图中间正文可见 Agent 在规划：

> 把 Agent / 任务 / 工具从三列收敛成「任务首页 + 具体任务页」…

说明 Codex 自身也在迭代 IA；但即便在「三列」阶段，它仍坚持：

- 中列 = **任务叙事**  
- 右列 = **结构化状态（Progress/Git）**  
- 左列 = **项目下任务列表**  

AWB 若只学「三列布局」而不学「列的职责」，会复制杂乱而非复制体验。

## 五、差距优先级（仅任务对话域）

### P0 — 改认知模型（不加大功能也能改善）

1. **主标题改为任务标题**（`Session.title` 用摘要/首句意图生成，禁止 `ls -l` 级标题）  
2. **主区首屏展示任务状态行**：进行中 / 已工作 Xm / 等待输入  
3. **工具结果默认摘要行**，终端/Diff 折叠；禁止全量 stdout 作为默认气泡  
4. **输入框文案任务化**：「继续说明本次任务的修改…」  
5. **顶栏产品 Tab 退出任务页**或收起到菜单  

### P1 — 对齐 Codex 监督感

6. **右栏（或可折叠抽屉）Progress**：从 Agent 计划或 tool 序列推导步骤清单  
7. **活动流组件**：`Read` / `Edited` / `Ran` 一行一条，与叙事段落分离  
8. **侧栏按项目/工作区分组**（`workspace` 或仓库名作为分组键）  
9. **会话列表显示耗时** `22m` 与状态点（执行中/待审批/完成）  

### P2 — 任务生命周期闭环

10. **Task 实体**（或在 Session 上扩展 `phase`、`stepsJson`、`elapsedMs`）  
11. **异步任务**：发送后可切换会话，侧栏保留 spinner  
12. **变更审查 + Git 摘要**（右栏 Git 区）  
13. **审批队列**（右栏或底部，非仅 bash 弹窗）

## 六、目标体验草图（对齐 Codex，保留 AWB 多 Agent 特色）

```
┌─────────────┬──────────────────────────────────┬──────────────┐
│ + 新任务     │  整理 Q1 销售数据 · 数据助手      │ 进度         │
│ 搜索        │  已运行 4m · 执行中              │ ☑ 读取表格   │
│             │                                  │ ☐ 生成图表   │
│ myagent ▾   │  我会先读取 uploads 下的 xlsx…   │ 工作区       │
│  · 整理… 4m │  Read sales.xlsx                │ myagent/     │
│  · 天气 昨天│  Ran 聚合脚本                     │ 本地 · 已连接 │
│             │                                  │ 待审批 (0)   │
│ 全部 Agent ▾│  继续本任务…              [发送] │              │
└─────────────┴──────────────────────────────────┴──────────────┘
```

与 Codex 的差异保留点：

- **多 Agent** 通过「助手名在标题旁」或分组体现，而非 Codex 单助手  
- **Hub 安装 Agent** 不进任务页顶栏，进「全部 Agent」折叠区

## 七、结论

**任务对话 UX 的差距，本质不是「我们有没有工具调用」，而是「界面是否围绕 Task 组织叙事与监督信息」。**

Codex 客户端让用户始终处于三种安心之中：

1. **我知道我在办哪件事**（任务标题 + 项目）  
2. **我知道进展到哪了**（耗时 + Progress + 活动摘要）  
3. **我知道要不要介入**（右栏状态 + follow-up 输入 + 权限在手边）  

Agent Workbench 当前让用户处于另一种状态：

1. 我知道在和哪个 **Agent 产品** 说话  
2. 我看到 **原始执行日志**  
3. 我要自己从聊天里 **猜任务是否结束**  

从 Codex 对齐的捷径不是再加一个终端面板，而是：

> **把 Session 呈现成 Task，把 ToolCall 呈现成 Activity，把 Progress/Git 呈现成右栏；把聊天降格为「任务叙事流」中的一种内容，而不是唯一内容。**
